Quick Start

Five steps from zero to your first results.

1
Launch the server

Double-click run.bat (Windows), or activate the virtual environment and run python app.py. Then open http://127.0.0.1:5000 in your browser.

2
Stage 1 — Configure

Select an AI provider, paste your API key, choose a model, and click Test Connection. If it shows a green check, you're ready.

3
Stage 2 — Ingest & Screen

Upload a reference file (.ris, .bib, .csv) exported from PubMed, Scopus, or Web of Science. Parse it, deduplicate, then optionally run AI abstract screening.

4
Stage 3 — Full-Text Processing

Upload your PDF papers, set your screening prompt and extraction fields, then click Start Processing. Watch live progress in the monitor.

5
Stage 4 — Results & Export

Review AI decisions, filter by Include / Exclude / Flag, click rows for full reasoning, and export to .xlsx.

Tip: Your settings (provider, API key, model, criteria) are saved automatically between sessions.

AI Providers

Nine providers are supported. All require an API key except Ollama.

Ollama Free

Runs entirely on your computer — no API key, no cost, no data leaves your machine. Requires 8 GB RAM. Best for privacy-sensitive research.

Setup: Install from ollama.com/download, run ollama pull llama3.2, then ollama serve. Set Base URL to http://localhost:11434.

DeepSeek Free credits

Very affordable. Free credits on sign-up; ~$0.01 per 1,000 abstracts after that. Great for large reviews on a budget.

Setup: platform.deepseek.com → API Keys → Create. Model: deepseek-chat.

Google Gemini Free tier

Generous free tier (1,500 requests/day on Gemini 2.0 Flash). Excellent for getting started without any payment method.

Setup: ai.google.dev → Get API key. Model: gemini-2.0-flash.

OpenAI Paid

Industry standard. New accounts receive a small free credit. Recommended model: gpt-4o-mini — highly capable and inexpensive (~$0.09 per 1,000 abstracts).

Setup: platform.openai.com → API keys → Create.

Anthropic (Claude) Paid

Excellent reasoning quality. Recommended for accuracy-critical extractions. Model: claude-3-5-haiku-20241022 (fast and affordable) or claude-3-5-sonnet-20241022 (best quality).

Setup: console.anthropic.com → API Keys.

Mistral Free tier

European provider with a free tier. Good for multilingual reviews. Model: mistral-small-latest.

Setup: console.mistral.ai → API Keys.

Kimi (Moonshot) Free credits

Good for reviews involving Chinese-language papers. Model: moonshot-v1-8k.

Setup: platform.moonshot.cn → API Management.

Grok (xAI) Paid

Latest xAI technology. Model: grok-beta.

Setup: console.x.ai → API Keys.

Custom OpenAI-Compatible Varies

Connects to any OpenAI-compatible server: LM Studio, vLLM, Together AI, Groq, etc. Enter the Base URL of your local or remote server.

Custom model IDs: If a model isn't in the dropdown (e.g., a newly released version), type its ID directly in the custom model input below the dropdown — it overrides the selection.

Stage 1 · Configure

Set up your AI provider before anything else. Settings are saved automatically.

AI Provider
Select your provider from the dropdown. Each provider has different pricing, capabilities, and setup requirements. See the AI Providers tab for details.
API Key
Paste your API key from the provider's dashboard. The key is stored only in webapp_settings.json on your own machine — it is never transmitted anywhere except directly to the AI provider when you run a request.
Model
Select from the predefined list for your provider. Smaller models (e.g., gpt-4o-mini) are faster and cheaper; larger models produce higher-quality decisions. For most abstract screening tasks, a small/medium model is sufficient.
Custom Model ID
Type any model identifier in the text box below the dropdown to override the selection. Useful for newly released models not yet in the list (e.g., gpt-4.5, claude-opus-4, llama3.3:70b). Leave blank to use the dropdown selection.
Base URL
Only shown for Ollama (Local) and Custom OpenAI-Compatible. Default for Ollama: http://localhost:11434.
Test Connection
Always test before running a large batch. A green result confirms the API key is valid and the model is accessible.

Stage 2 · Ingest & Screen

Import references from databases, remove duplicates, and screen abstracts with AI.

Step A — Upload Reference File

Export your search results from academic databases and upload the file here.

FormatExtensionExport from
RIS.risPubMed (NBIB → RIS), Scopus, Web of Science, Embase, Zotero
BibTeX.bibWeb of Science, Google Scholar, Mendeley, Zotero
CSV.csvScopus, Web of Science (must include Title, Abstract, DOI columns)

Step B — Parse & Deduplicate

Parse: Extracts title, authors, year, abstract, journal, and DOI from the file.

Deduplicate: Removes duplicates in two passes — exact DOI matches first, then fuzzy title matching (90% similarity threshold). This eliminates papers that appear in multiple databases.

Step C — AI Abstract Screening

Enter your inclusion/exclusion criteria in the text area, then click Screen Abstracts. The AI reads each abstract and returns one of:

  • Likely Include — abstract meets your criteria
  • Likely Exclude — abstract does not meet criteria
  • Flag for Review — AI is uncertain; you must decide

You can stop and resume screening at any time. Results appear in the table below as they come in.

Tip: Abstract screening is optional — you can skip directly to Stage 3 and upload PDFs if you prefer to screen at the full-text level.

Stage 3 · Full-Text Processing

Upload PDFs for AI-powered screening and structured data extraction.

Upload PDFs

Drag and drop PDF files onto the upload zone, or click to browse. You can upload many files at once. The tool extracts text from each PDF and sends it to the AI.

Processing Options

Parallel Processing
When enabled, multiple PDFs are processed simultaneously, significantly reducing total time. Disable only if you experience rate-limit errors.
Max Workers
Number of parallel processes (default: 3). Each worker uses a separate API call at the same time. Start with 2–3; increase if your API plan allows higher rate limits.
Rate Delay (s)
Pause in seconds between sending papers to the AI (default: 1.0). Increase to 2.0–3.0 if you see "Too Many Requests" errors.
Two-Stage Screening
The AI first does a fast abstract-level check. Only papers that pass this check proceed to full extraction. Saves significant cost when many papers are expected to be excluded.
Enable Caching
Skips files already successfully processed in a previous run. Always keep this enabled. If processing is interrupted, you can restart without re-paying for completed files.

Screening Prompt & Extraction Fields

Screening Prompt: Your inclusion/exclusion criteria written in plain language. The AI applies these to the full text of each paper. See the Writing Criteria tab for examples.

Extraction Fields: One field name per line. The AI extracts this information from each included paper. Examples: title, authors, study_design, sample_size, main_findings.

Processing Monitor

While processing runs, the monitor shows progress, decision counts (included / excluded / flagged), files per minute, and a live log. You can safely stop processing at any time — cached results are preserved.

Writing Criteria

Well-written criteria are the most important factor in review quality.

Inclusion Criteria — Example (Medical)

INCLUSION CRITERIA:
• Study Type: Randomized controlled trials (RCTs) or quasi-experimental studies
• Population: Adult human participants (age 18+)
• Intervention: Any pharmacological or behavioral intervention for type 2 diabetes
• Outcome: At least one primary outcome measuring glycemic control (HbA1c, fasting glucose)
• Publication Year: 2015 to 2025
• Language: English only
• Publication Type: Peer-reviewed journal articles

Exclusion Criteria — Example

EXCLUSION CRITERIA:
• Animal studies or in vitro studies
• Review articles, systematic reviews, or meta-analyses
• Conference abstracts without full text
• Studies with fewer than 30 participants
• Studies without a control group

Tips for Effective Criteria

  • Be specific — "RCTs only" is better than "good quality studies."
  • Use standard terminology — RCT, cohort, cross-sectional, meta-analysis. The AI knows these terms.
  • Separate inclusion from exclusion — never mix them in the same block.
  • Test with known papers — find 5 papers that should include and 5 that should exclude, run them, verify the AI agrees.
  • Iterate — if the AI makes systematic errors, refine the relevant criterion.
  • Write for a research assistant — imagine explaining your criteria to a knowledgeable but uninformed assistant.

How Pricing Works

AI providers charge per token — roughly 4 characters, or ¾ of a word. You pay for both what you send (input) and what the AI returns (output). Input tokens are usually cheaper.

Tips to Minimize Costs

  • Always enable caching — prevents re-charging for files already processed.
  • Test with 5–10 papers first — verify your criteria work before running 3,000 papers.
  • Use small models for abstract screening — gpt-4o-mini or deepseek-chat are sufficient for title/abstract decisions.
  • Use larger models for extraction only — if you need high quality extraction, upgrade only for the full-text stage.
  • Enable Two-Stage processing — screens quickly first, extracts only from papers that pass, saving tokens on excluded papers.
  • Watch the token counter — the monitor shows running token usage in real time.

FAQ & Troubleshooting

My API key isn't working / "Authentication failed"

1. Log into your provider's website and confirm the key is still active.
2. Re-copy the key — hidden spaces at the start or end are a common cause.
3. Check your account has credit or a payment method on file.
4. Use Test Connection to see the exact error message.

"Rate Limit Error" or "Too many requests"

You are sending papers faster than your API plan allows.
1. Increase Rate Delay to 2.0 or 3.0 seconds.
2. Reduce Max Workers to 1 or 2.
3. Wait 5–10 minutes for the rate limit to reset.
4. Upgrade your API plan for sustained high-volume work.

Ollama connection failed

1. Open a terminal and run ollama serve. Leave the window open.
2. Confirm the Base URL is http://localhost:11434.
3. Verify a model is downloaded: run ollama list.
4. If no models appear, run ollama pull llama3.2.

The AI is making wrong screening decisions

This almost always means the criteria need refinement.
1. Read the Reasoning column — the AI explains why it made each decision.
2. Identify the pattern in wrong decisions (e.g., including animal studies you want excluded).
3. Add that condition explicitly to your exclusion criteria.
4. Re-run on the problematic subset.

PDF files show "Error" in results

The PDF likely contains scanned images without a text layer (no OCR).
1. Open the PDF — if you cannot copy text from it, it is image-only.
2. Use Adobe Acrobat or an online OCR tool to add a text layer.
3. Password-protected PDFs also cannot be processed — remove the password first.

Processing stopped unexpectedly

1. Check the processing log for the last error message.
2. Verify your internet connection.
3. Re-click Start Processing — caching means completed files are skipped automatically.
4. If RAM is low, reduce Max Workers to 1.

Reference file fails to parse

1. Confirm the file is a valid .ris, .bib, or .csv.
2. For CSV: ensure it has columns named title and abstract (or Title / Abstract).
3. Some databases export CSV with different column names — rename them in Excel before uploading.
4. For RIS files from PubMed, use the "NBIB" format and rename the extension to .ris.

Settings are not saved between sessions

Settings are saved to webapp_settings.json in the WebApp folder. Check that the file is not in a read-only location. If the file exists but is empty, delete it and restart the server.

Can I use this tool for multiple review projects?

Yes. Each time you start a new project, clear the uploads folder or restart the server (which resets the in-memory session). Your screening criteria and provider settings persist in webapp_settings.json. You can also keep separate copies of the WebApp folder for different projects.

How do I cite this tool in my paper?

Cite the tool as: Anisi, M. (2026). SLR Automation Tool (v3.3.0). GitHub. https://github.com/sadeghanisi/SLR
Also note the AI model and provider used, the date of the screening run, and your inclusion/exclusion criteria in your methods section.

Important Notice

What This Tool Is — and Is Not

This tool IS
  • An automation assistant that accelerates the mechanical, time-consuming parts of a systematic review — uploading, parsing, deduplication, and initial screening passes.
  • A first-pass filter. It reduces the volume of papers a human must read, not the need for human reading.
  • A data-extraction helper that produces a structured first draft. The researcher is responsible for verifying every extracted value.
  • A research productivity tool intended to complement, not replace, rigorous methodological judgment.
  • Open-source software provided under the MIT License — free to use, modify, and distribute with attribution.
This tool IS NOT
  • Not a peer reviewer or substitute for expert methodological assessment of study quality.
  • Not infallible. AI models make classification errors, miss nuance, misread ambiguous text, and may hallucinate extracted values. Error rates vary by model, domain, and criteria complexity.
  • Not a guarantee of PRISMA compliance. Compliance requires human judgment, documented search strategies, risk-of-bias assessment, and proper reporting — none of which this tool performs automatically.
  • Not a substitute for human screening. All AI decisions should be treated as recommendations pending researcher verification.
  • Not affiliated with OpenAI, Anthropic, Google, or any other AI provider. It is a third-party client that calls their public APIs.
  • Not validated for clinical, regulatory, or policy decision-making contexts without independent verification of outputs.

Your Responsibility as the Researcher

By using this tool you accept full responsibility for the quality, accuracy, and integrity of your systematic review. Specifically:

  • Verify every AI decision. Screen a representative sample of AI-classified papers manually — including both included and excluded papers — to estimate the error rate before trusting the full output. Best practice is to verify at least 10–20% of decisions independently.
  • Verify every extracted value. Data extracted from PDFs must be checked against the source paper before inclusion in your review table or analysis. AI extraction is a first draft, not a final record.
  • Maintain a complete audit trail. Document the exact AI provider, model name and version, API access date, inclusion/exclusion criteria text, and any post-hoc corrections you made. This is required for transparent reporting.
  • Do not rely solely on AI for eligibility decisions. Final inclusion and exclusion decisions for a PRISMA-compliant review must involve human judgment, ideally with dual independent screening for a subset of records.
  • Understand your criteria before automating. If your inclusion/exclusion criteria are ambiguous to you, they will produce unpredictable AI results. Pilot-test on 10–20 papers with known outcomes before running a full batch.

Academic & Scientific Integrity

  • Disclosure is mandatory. You must disclose the use of AI-assisted screening in your methods section. Failure to do so may violate journal policies, institutional guidelines, and research integrity standards.
  • Cite the tool correctly. When reporting, cite the tool, the AI model used, and the access date. Example methods statement: "Abstract screening was conducted using SLR Automation Tool v3.3.0 (Anisi, 2026) with [Provider] [Model], with manual verification of a random 15% sample (Cohen's κ = 0.XX)."
  • AI outputs are not citable sources. AI-generated reasoning in the results panel is a processing artifact, not a peer-reviewed finding. It must not be quoted or presented as evidence in your review.
  • Flagged papers require human decision. Papers marked "Flag for Review" have been identified as requiring human judgment. Every flagged paper must be reviewed manually before a final inclusion/exclusion decision is recorded.
  • Cross-database deduplication is not exhaustive. The fuzzy-matching deduplication in this tool catches most, but not all, duplicates. A final manual check of your included papers list is recommended.
  • Model bias. AI language models are trained on existing literature and may exhibit systematic biases — for example, favouring studies from certain languages, regions, or publication types. Be aware of this when interpreting results, particularly for under-represented populations or languages.

  • Costs can escalate unexpectedly. Processing a large folder of PDFs or running multiple workers simultaneously can consume significant tokens. Always test with a small batch first. Use caching to prevent accidental re-processing.
